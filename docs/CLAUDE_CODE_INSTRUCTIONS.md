# 回顧録(kaikoroku)— Claude Code on the Web 継続開発指示書

## 前提
- Phase 1 は完成済み(このリポジトリのコード)。React + TypeScript + Vite + PWA。
- 予想の記録・一覧・詳細・編集・削除、IndexedDB永続化、PWA化まで動作確認済み。
- **開発はスマホの Claude Code on the Web(GitHubリポジトリ連携)で行う。**
  ローカルPC・Macは使わない前提。デプロイはVercelにリポジトリ接続で自動化。

## このアプリが勝つ理由(すべての実装判断の北極星)
- 競合(netkeiba/umanote/pakara)は「情報を見る」「メモを書く」アプリ。
  本アプリは「予想の仮説を記録し、結果と突き合わせて検証する」唯一のアプリ。
- 差別化の心臓は3つ:①AIによる入力の構造化 ②仮説単位の答え合わせ判定 ③見送り判断の記録。
- 非機能の勝ち筋:「軽い・落ちない・熱くならない」。現状gzip約83KB。この軽さを守ること。

## 絶対に作らないもの(スコープ外・追加提案も不要)
- レースデータ/出馬表/オッズの取得・表示(データ許諾リスク回避のため設計思想として排除)
- 他ユーザーの予想閲覧などのSNS機能/馬券購入連携/凝った収支グラフ
- **AIが独自に買い目・予想を生成する機能**(審査・法務リスク。将来のv2でも
  「ユーザー自身の過去データの自己参照」のみ許可。AIの独自予測は永久に禁止)

## リポジトリの設計ルール(必ず守る)
- データアクセスは `src/repository.ts` の repository 層に隔離済み。
  DB実装を差し替えるときは `PredictionRepository` インターフェースを維持すること。
- スキーマは `src/types.ts`。変更時はSupabaseテーブルとの整合を保つ。
- デザイントークンは `src/styles.css` の `:root`。新画面もこのトークンから作る。
- 各Phaseの完了ごとに、Vercelプレビューで動作確認できる状態でコミット・PRすること。

---

## Phase 1.5|Supabase同期
1. Supabaseプロジェクト作成手順を提示(ユーザーがスマホのブラウザで実施)
2. `predictions` テーブルをsrc/types.tsのスキーマ通りに作成するSQLを出力
3. Supabase Auth(メールマジックリンク)でログイン導線を追加
4. `src/repository.ts` にSupabase実装を追加し、ログイン時はSupabase・未ログイン時はローカルを使う
5. ローカル→Supabaseへの初回データ移行(アップロード)機能
- 環境変数(SUPABASE_URL, ANON_KEY)はVercelの環境変数で管理。コードに直書き禁止。

## Phase 2|AI構造化・AI答え合わせ(心臓部)
1. Vercel Functions(/api)に中継エンドポイントを2本作る:
   - POST /api/structure … 予想の殴り書き→5項目JSON(モデル: claude-haiku-4-5)
   - POST /api/judge … 予想+結果→仮説別判定JSON(モデル: claude-sonnet-4-6)
   - ANTHROPIC_API_KEY はVercel環境変数。クライアントに露出させない。
   - システムプロンプトは prompt caching を有効化(固定文のため)。
2. システムプロンプトは本リポジトリの docs/prompts.md の3本を一字一句そのまま使う
   (補完禁止・結果論で裁かない等の制約が製品の生命線。勝手に変えない)。
3. 予想入力画面に「殴り書きモード」を追加:フリーテキスト(+可能ならWeb Speech API、
   非対応環境はキーボード音声入力への案内)→ /api/structure → 5項目に反映→ユーザー確認→保存。
4. 詳細画面に「結果入力」→ /api/judge → 仮説別バッジ(的中/一部/外れ/検証不能)+
   コメント表示。**判定はユーザーが1タップで上書き可能にする(AIは下書き、確定は人間)。**
5. レート制御:1ユーザー月150回のAPI呼び出し上限(Supabase側でカウント)。
   入力20字未満はAPIを呼ばずクライアント側で弾く。

## Phase 3|馬場ノート
- 開催日×競馬場×コース(芝/ダート)単位のノート。/api/structure-track(haiku)でタグ構造化。
- 予想入力画面で同競馬場の直近ノートを自動参照表示(これがこの機能の存在意義)。

## Phase 4|検証ノート(資産画面)
- 検証済み一覧+集計:5項目別の仮説的中率、競馬場別・券種別、見送り判断の正解率。
- グラフはシンプルな棒1種のみ。ライブラリを増やさず軽さ優先(SVG手描きで可)。

## Phase 5|Stripe課金
- サブスク:無料トライアル7日→月480円/年3,800円。Stripe Checkout + Customer Portal。
- トライアル後:閲覧のみ可・新規記録不可(過去データは読める)。
- Webhook(Vercel Functions)でSupabaseの課金状態を同期。

## 進め方のルール
- 各Phaseは小さなタスクに分割し、1タスク=1セッション=1PRで進める。
- 要件・設計の合意事項は docs/ ディレクトリに追記し、コードと同期させる。
- 各PRの説明に「Vercelプレビューでの確認手順」を必ず書くこと。
- ユーザーは非エンジニア。専門用語は使ってよいが、ユーザーに操作を求めるときは
  スマホで実行できる手順に限定すること。
